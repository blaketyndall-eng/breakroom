# Claude Design Handoff

This document describes how to integrate design files delivered by Claude into the Breakroom project.  It acts as a bridge between the Claude design output and our Astro + React + Tailwind implementation.

## Design Zip Structure

When you receive a ZIP of designs from Claude (likely containing HTML, CSS, images, and annotations), extract it to the following directory inside the project:

```
/design/claude/
  screenshots/
  html/
  css/
  notes/
  assets/
```

Where:

- `screenshots/` holds reference images for each page or component.
- `html/` contains HTML prototypes or codepen exports generated by Claude.
- `css/` contains CSS files used in the prototypes.
- `notes/` stores text files with annotations, comments, or design rationale.
- `assets/` stores image assets (icons, photos) used in the design.  All image names should be descriptive.

## How to Implement Designs

1. **Review the screenshots and HTML prototypes**:  They show how Claude envisions the page layout.  Examine structure, spacing, and alignment.  Identify objects (motel keys, clocks, dice) and UI elements (buttons, links, forms).

2. **Identify reusable components**:  Each design should map to a component in `src/components`.  For example:
   - A search bar → `SearchBox.tsx`.
   - A bracket diagram → `Bracket.tsx`.
   - A player card → `PlayerCard.tsx`.
   - A product row → `ProductRow.tsx`.
   - A lore object card → `ObjectCard.astro`.

3. **Extract styling information**:  Claude’s HTML may use inline styles or external CSS.  Convert these into Tailwind classes or create small utility classes in your CSS modules.  Respect the colour palette and type hierarchy defined in `STYLE_GUIDE.md`.

4. **Integrate icons and images**:  Place images referenced in Claude’s design into `public/images/` or `src/assets/`.  Use PptxGenJS guidelines if these images appear later in slides.  Reference images in components via `addImage` (for slides) or `<img>` tags (for web).

5. **Preserve layout authenticity**:  The old‑web aesthetic is intentional.  Recreate table layouts, bordered modules, and link styling.  Use the utility classes defined in `styles/oldweb.css` where applicable.

6. **Map interactions**:  If the prototype includes interactive behaviours (hover effects, modals, hidden links), implement these in React islands.  Keep interactions minimal and accessible.  Use `useState`, `useEffect`, and simple event handlers.

7. **Annotate differences**:  If the design conflicts with the Style Guide or Route Map, document the difference in this file and discuss with the project lead.  Do not blindly copy over colours or fonts if they break consistency.

8. **Incorporate lore**:  Ensure that every object or label in the design has a corresponding entry in the Lost & Found or data model.  Use the names and icon references consistently across pages.

## Collaboration Workflow

1. **Create a branch** for each page or component you’re implementing from Claude’s design:  `feature/claude-<component>`.

2. **Commit the design assets** (screenshots, HTML, CSS) into `design/claude/` within your branch.  Do not leave these files in `/src` or `/public`.

3. **Use the design as a reference** while coding.  Do not copy the entire HTML structure verbatim; convert to React/Astro and Tailwind.

4. **Add comments** in your component code referencing the design (e.g., `// From Claude design: see design/claude/html/idle-hands.html`).

5. **Open a Pull Request** to the main dev branch when the component or page matches the design.  Include before/after screenshots and note any deviations or design decisions.

6. **Iterate** based on feedback from the project lead.  Merge once approved.

This handoff document should be updated whenever we receive a new batch of design assets.  Keep the design folder organised and make it easy for other developers to map designs to components.
